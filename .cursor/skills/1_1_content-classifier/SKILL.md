# Content-Classifier — Анализ и классификация исходников

## Inputs

- [project-state.json](../../../project-state.json)
- Сконвертированные `.md` файлы в [sources/converted/](../../../sources/converted/)

## Logic

### Шаг 3: Чтение и анализ

Прочитай каждый `.md` файл в [sources/converted/](../../../sources/converted/).

> На этом этапе ты ещё не знаешь, про что будет курс. Ты изучаешь и классифицируешь то, что есть, чтобы сделать отчёт с предположением.

**Стратегия чтения в зависимости от объёма файла:**

| Объём | Стратегия | Пометка |
|-------|-----------|---------| 
| ≤ 5 000 слов | Прочитать полностью | `scan_mode: full` |
| > 5 000 слов | Режим «scan & summarize» (см. ниже) | `scan_mode: headers_only` |

**Режим «scan & summarize» (для файлов > 5 000 слов):**
1. Прочитать первые 200 строк целиком (вводная часть, оглавление)
2. Извлечь все заголовки H1–H3 по всему файлу
3. Прочитать по 1 абзацу после каждого заголовка (суть раздела)
4. Прочитать последние 50 строк (заключение, выводы)

Для каждого файла зафиксируй:
- Имя файла (включая подпапку, если есть)
- Приблизительный объём (количество слов)
- Режим сканирования (`full` / `headers_only`)
- Качество конвертации (из лога конвертера, если доступно, либо оцени по тексту)
- Ключевые разделы (заголовки H1–H3)
- Краткое описание содержимого (2–3 предложения)

**Проверка дедупликации:**

После анализа всех файлов проверь пересечения по содержимому. Если обнаружены — покажи **краткую** сводку с приоритизацией:

> «Обнаружены пересечения:
> 🔴 **Критичные** (один документ полностью дублирует другой):
> - `X.md` ≈ `Y.md` — оба содержат [тему целиком]. Нужно выбрать один.
>
> 🟡 **Незначительные** (частичное пересечение в отдельных разделах):
> - `A.md` и `B.md` — общий раздел «[раздел]». Можно оставить оба.
>
> Какие файлы актуальнее?»

**Правила приоритизации:**
- **Критичные** — файлы дублируют друг друга на ≥70% (одни и те же разделы, аналогичный текст). Обязательно решить, какой оставить.
- **Незначительные** — частичное совпадение (1–2 раздела). Информировать, но не блокировать.

Дождись ответа пользователя только по критичным пересечениям. Незначительные — зафиксируй в отчёте и продолжай.

### Шаг 4: Классификация контента

Двухуровневая классификация каждого файла.

#### Уровень 1 — бинарная классификация

Определи тип каждого файла:

**«Исходный материал»** — признаки:
- Текст — статья, регламент, инструкция, политика, расшифровка
- Нет разбивки на слайды, экраны или интерактивные элементы
- Линейный материал без пометок «Экран», «Слайд», «Квиз»

**«Сценарий курса»** — признаки:
- Есть разбивка на экраны/слайды/страницы
- Присутствуют пометки интерактивов: квизы, аккордеоны, matching, scenario
- Видна методическая структура: вступление → теория → практика → итог

#### Уровень 2a — подтип исходного материала

Если файл определён как «Исходный материал», определи подтип:

| Подтип | Ключ | Признаки |
|--------|------|----------|
| ВНД / регламент | `regulatory` | Политики, процедуры, SOP, приказы, инструкции, нормативные документы |
| Презентация | `presentation` | Слайды для обучения или совещаний, тезисный формат |
| Расшифровка | `transcript` | Расшифровки звонков, интервью, фокус-групп, вебинаров, лекций |
| Наброски / планы | `draft` | Черновики, заметки, наброски планов курса, mind-maps |
| Справочник | `reference` | Каталоги, справочные таблицы, глоссарии, технические спецификации |
| Статья / аналитика | `article` | Статьи, обзоры, исследования, аналитические отчёты |

#### Уровень 2b — шкала готовности сценария

Если файл определён как «Сценарий курса», оцени его готовность по шкале 0–100%:

| Диапазон | Что означает |
|----------|-------------|
| 0–30% | Есть структура (разбивка на экраны), но нет контента |
| 31–60% | Есть контент, но нет интерактивов/квизов или есть значительные пробелы |
| 61–90% | Почти готов, но есть отдельные пробелы |
| 91–100% | Полностью готовый сценарий с интерактивами |

**Критерии оценки готовности:**
- **+20%** за наличие структуры (модули, экраны/страницы)
- **+20%** за наполненность текстом (не плейсхолдеры)
- **+20%** за интерактивные элементы (квизы, задания)
- **+20%** за методическую логику (цели → теория → практика → итог)
- **+20%** за полноту (нет пустых экранов, нет пометок «TODO»)

### Шаг 5: Запись отчёта в [workspace/sources-report.md](../../../workspace/sources-report.md)

Запиши результат анализа в файл [workspace/sources-report.md](../../../workspace/sources-report.md).

> **Структура отчёта: сначала общая картина, потом детали.**

Шаблоны и примеры отчётов находятся в папке [assets/](assets/):
- [report-template-full.md](assets/report-template-full.md) — шаблон отчёта при наличии исходников
- [report-template-none.md](assets/report-template-none.md) — шаблон отчёта без исходников
- [report-example-full.md](assets/report-example-full.md) — пример заполненного отчёта
- [report-example-none.md](assets/report-example-none.md) — пример заполненного отчёта без исходников

**Ключевые правила наполнения отчёта:**

1. **Frontmatter** — заполни YAML-блок в начале файла. Это машиночитаемые данные для downstream-этапов.
2. **Общая оценка** — сначала опиши все материалы в совокупности: впечатление, качество, структура.
3. **Гипотеза о тематике проекта** — на основе содержимого файлов сформулируй предположение, какой теме предположительно посвящён проект. Это именно гипотеза — окончательная тема определяется на этапе брифования.
4. **Детальные карточки** — затем опиши каждый файл отдельно: содержимое, тип, подтип, сильные/слабые стороны.
5. **Дубликаты** — если были обнаружены пересечения, зафиксируй решение пользователя.

## Outputs

- Файл [workspace/sources-report.md](../../../workspace/sources-report.md) — структурированный отчёт с YAML frontmatter
