# Generation-Text — Генерация учебных текстов

## Inputs

- [project-state.json](../../../project-state.json) (статус проекта)
- [workspace/brief.md](../../../workspace/brief.md) (редполитика, тон, аудитория, принципы проектирования)
- [sources/converted/*.md](../../../sources/converted/) (исходный контент — для блоков с `content_basis: source`)
- [content/pages/*.yaml](../../../content/pages/) (шаблоны с плейсхолдерами — основной рабочий материал)

> **Важно:** Этот скилл НЕ читает [workspace/course-structure.yaml](../../../workspace/course-structure.yaml) и НЕ читает спецификацию модели [`2_1_course-model`](../2_1_course-model/SKILL.md). Вся необходимая метаинформация (purpose, source_ref, content_basis) уже встроена в каждый блок файла страницы скиллом `course-template`.

> **Этот скилл работает последовательно** — не через субагентов. Один агент обрабатывает страницы по очереди **партиями (батчами)**, что обеспечивает единую терминологию, стиль и нарратив, при этом не перегружая контекстное окно на больших курсах.

## Logic

### Шаг 0: Проверка статуса проекта

Прочитай [project-state.json](../../../project-state.json).

1. Убедись, что скилл `course-template` выполнен.
2. Проверь `_active`: если установлен другим скиллом — предупреди пользователя и не начинай без подтверждения. Если `_active` от предыдущего прерванного запуска этого же скилла — подхвати работу (см. «Восстановление после сбоя»).
3. Составь список всех файлов [content/pages/*.yaml](../../../content/pages/), у которых есть presentation-блоки со `status: empty` нужных типов. Это **pages_to_process**.
4. Установи `_active`:

```yaml
_active:
  skill: generation-text
  started: "{ISO timestamp}"
  pages_total: {len(pages_to_process)}
  pages_done: 0
  batch_size: 10
  current_batch: 1
```

### Шаг 1: Подготовка контекста

Прочитай из [brief.md](../../../workspace/brief.md):
- Tone of voice (стиль, обращение, юмор)
- Бизнес-драйвер и принципы проектирования
- Целевую аудиторию

> **Важно:** Этот скилл не задаёт пользователю вопросов по контенту. Все вопросы о достаточности источников и разрешениях на генерацию уже решены на этапе `course-matrix` (шаг «Верификация источников контента»). Поля `content_basis` и `source_ref` к этому моменту гарантированно валидны.

### Шаг 2: Генерация текстов партиями

Раздели **pages_to_process** на батчи по **10 страниц**. Обрабатывай батчи последовательно.

#### Глоссарий терминов

Перед первым батчем создай пустой глоссарий — список ключевых терминов и их каноничных написаний. После каждого батча **дополняй** глоссарий новыми терминами, встретившимися при генерации. В начале следующего батча перечитай глоссарий, чтобы гарантировать консистентность терминологии даже при сбросе контекста.

Глоссарий сохраняется в [workspace/_glossary.yaml](../../../workspace/_glossary.yaml):

```yaml
terms:
  - term: "двухфакторная аутентификация"
    not: ["двухэтапная верификация", "2FA верификация"]
    first_used: "page 2"
  - term: "инцидент информационной безопасности"
    not: ["ИБ-инцидент", "случай нарушения"]
    first_used: "page 5"
```

#### Навигация по структуре страницы

Файлы страниц имеют иерархию: `page.scene.frames[].blocks[]`. Блоки также находятся внутри вложенных сцен (`nested_scene.frames[].blocks[]`) и элементов блоков с вложенной структурой (`accordion.items[].blocks[]`, `tabs.items[].blocks[]`, `carousel.items[].blocks[]`).

**Алгоритм поиска блоков для заполнения:**
1. Обойти все фреймы в `page.scene.frames`
2. В каждом фрейме: обработать `blocks[]`
3. Если у фрейма есть `nested_scene` — рекурсивно обойти его фреймы
4. Для блоков `accordion`, `tabs`, `carousel` — обойти `items[].blocks[]`

#### Обработка батча

Для каждой страницы в батче (по порядку):

1. Прочитай файл страницы
2. Обойди все фреймы и найди presentation-блоки со `status: empty` следующих типов:
   - `tinymce` — текстовый блок
   - `accordion` — заполнить `items[].title` и вложенные блоки
   - `tabs` — заполнить `items[].title` и вложенные блоки
   - `carousel` — заполнить вложенные блоки в `items[]`
   - `flipcard` — заполнить `cards[].front` и `cards[].back`
   - `timeline` — заполнить `events[].date`, `events[].title`, `events[].description`
   - `quote` — заполнить `content` (и `author`, `source` если применимо)
   - `note` — заполнить `content` (тип note уже задан в `note_type`)
   - `table` — заполнить `headers` и `rows`
3. Для каждого блока прочитай встроенные поля `purpose`, `source_ref`, `content_basis`
4. **Если `content_basis: source`** — прочитай раздел из [sources/converted/](../../../sources/converted/) по `source_ref` и переработай
5. **Если `content_basis: brief` или `generate`** — сгенерируй текст на основе `purpose`, брифа и domain knowledge
6. Запиши контент в соответствующие поля, обнови `status: empty` → `status: filled`

**После каждой страницы:**
1. Обнови `status` блоков в файле немедленно
2. Обнови `_active.pages_done` в [project-state.json](../../../project-state.json)

**После каждого батча:**
1. Обнови `_active.current_batch` в [project-state.json](../../../project-state.json)
2. Дополни [workspace/_glossary.yaml](../../../workspace/_glossary.yaml) новыми терминами из этого батча

**Восстановление после сбоя:** если при запуске обнаружен `_active` от предыдущего прерванного запуска — подхвати работу: прочитай [workspace/_glossary.yaml](../../../workspace/_glossary.yaml) (если есть), пропусти страницы, где все целевые блоки уже `filled`, начни с первой страницы с `empty`-блоками.

**Правила генерации:**
- Строго следуй tone of voice из [brief.md](../../../workspace/brief.md)
- `purpose` блока — главный ориентир: текст должен выполнять именно эту задачу
- Роль фрейма (`role`) задаёт контекст: intro → мотивирующий, concept → объясняющий, summary → резюмирующий
- Если есть `source_ref` — текст основывается на исходнике, не выдумывай факты
- Если `source_ref` нет — генерируй на основе `purpose` и брифа
- Не выдумывай конкретные цифры, даты, имена, если их нет в источниках или брифе
- Адаптируй сложность под уровень аудитории
- Для `tinymce`: формат — HTML (`<p>`, `<strong>`, `<ul>`, `<ol>`, `<h3>` и т.д.)
- **Консистентность:** используй единую терминологию по всем страницам (глоссарий)
- **Не обрабатывай:** `image`, `hotspot`, `video` и все assessment-блоки — они заполняются другими скиллами

**Ориентир по объёму** — бери из поля `content_volume` страницы:
- `small` — до 150 слов суммарно на все текстовые блоки страницы
- `medium` — 150–350 слов
- `large` — 350–600 слов

# Запрет на фабрикацию

При генерации контента курса:

- **Не выдумывай** конкретные цифры, статистику, проценты
- **Не выдумывай** даты, ФИО, названия организаций
- **Не выдумывай** номера ГОСТов, ISO-стандартов, статей законов, приказов
- **Не цитируй** конкретные нормативные документы, если их нет в [sources/converted/](../../sources/converted/)

Если данных нет в источниках или брифе — используй обобщённые формулировки:
- «согласно внутренней политике компании» вместо «согласно приказу №547»
- «по данным исследований» вместо «по данным исследования McKinsey 2024 года»
- «значительная часть инцидентов» вместо «78% инцидентов»

### Шаг 3: Завершение и автозапуск следующего этапа

После заполнения **всех** страниц:

1. Удали `_active` из [project-state.json](../../../project-state.json).
2. Обнови [project-state.json](../../../project-state.json): отметь `generation-text` → `completed`.
3. Кратко сообщи пользователю прогресс (1–2 строки): «Тексты готовы (N страниц, N блоков). Перехожу к генерации проверочных заданий...»
4. **Немедленно запусти** скилл `generation-quiz`. Не спрашивай подтверждения — это единая цепочка генерации.

## Outputs

- Обновлённые файлы [content/pages/*.yaml](../../../content/pages/) с заполненными presentation-блоками (`status: filled`)
- Обновлённый [project-state.json](../../../project-state.json)
- **Автоматический запуск** `generation-quiz`
