# Course-Model — Модель учебной страницы

> **Назначение.** Этот скилл — спецификация, а не пошаговый процесс. Он описывает, как устроены страницы курса на платформе. Скиллы проектирования (`course-structure`, `course-matrix`, `course-template`) опираются на эту модель при создании структуры и шаблонов.

> **Область применения.** `course-model` читается **только на этапе проектирования**. Generation-скиллы (`generation-text`, `generation-quiz`, `generation-image`) не обращаются к этой спецификации — вся нужная информация уже заложена в YAML-файлы страниц скиллом `course-template`.

**Версия модели:** `1.0`

---

## 1. Главная идея

Учебная страница — это **структурированная модель**, а не сплошной текст. Контент организован в иерархию сущностей, которая задаёт ритм прохождения, поддерживает вложенные сценарии и отделяет смысловую структуру от визуального слоя.

**Формула:**

> **Страница → Сцена → Фреймы → Блоки**
> при этом горизонтальные сцены и банки вопросов могут вкладываться во фреймы основной сцены.

---

## 2. Иерархия сущностей

```
страница
 └─ vertical_scene (ровно одна)
     └─ фрейм
         ├─ блоки (presentation / assessment)
         ├─ вложенная horizontal_scene (опционально)
         └─ банк вопросов (опционально)
```

### Формат ID

Все сущности ниже страницы используют единый формат ID:

| Сущность | Формат ID |
|----------|-----------|
| Страница | Целочисленный Primary Key (`1`, `2`, `3`, ...) |
| Сцена | 7-символьный hexadecimal (`a1b2c3d`) |
| Фрейм | 7-символьный hexadecimal (`b2c3d4e`) |
| Блок | 7-символьный hexadecimal (`c3d4e5f`) |
| Банк вопросов | 7-символьный hexadecimal (`d4e5f6a`) |

Все hex-ID уникальны в рамках всего курса.

### 2.1. Страница

Верхнеуровневая учебная единица. Содержит **ровно одну** `vertical_scene`.

**Обязательные поля:**

| Поле             | Описание                               |
| ---------------- | -------------------------------------- |
| `id`             | Целочисленный Primary Key              |
| `title`          | Название страницы                      |
| `slug`           | URL-совместимый идентификатор          |
| `goal`           | Учебная цель страницы                  |
| `estimated_time` | Ожидаемое время прохождения (минуты)   |
| `scene`          | Единственная `vertical_scene` страницы |

**Необязательные поля:**

| Поле            | Описание                                  |
| --------------- | ----------------------------------------- |
| `pass_criteria` | Условие зачёта страницы (см. раздел 8)    |

### 2.2. Сцена

Контейнер прохождения. Задаёт ритм, последовательность шагов и способ показа материала. Всегда состоит из фреймов.

**Два типа:**

| Тип                | Уровень      | Логика                                             |
| ------------------ | ------------ | -------------------------------------------------- |
| `vertical_scene`   | Верхний      | Фреймы идут сверху вниз — основной ритм страницы   |
| `horizontal_scene` | Вложенный    | Фреймы показываются как шаги/слайды                |

**Правило одной сцены:**
- На странице — **ровно одна** `vertical_scene`
- Внутри неё (во фреймах) может быть **несколько** вложенных `horizontal_scene`
- `horizontal_scene` не может быть верхнеуровневой сценой страницы
- Вложенность сцен — не более 2 уровней (vertical_scene → horizontal_scene)

Подробнее → [`assets/scene-types.yaml`](assets/scene-types.yaml)

### 2.3. Банк вопросов

Специальная вложенная сущность для серии упражнений или тестовых вопросов. Строится на фреймах — каждый фрейм = один вопрос или упражнение.

**Два типа:**

| Тип                        | Логика                                          | Когда использовать                                     |
| -------------------------- | ----------------------------------------------- | ------------------------------------------------------ |
| `vertical_question_bank`   | Вопросы идут вертикально сверху вниз            | Итоговые тесты, серьёзные проверки, длинные assessment  |
| `horizontal_question_bank` | Вопросы показываются как последовательность экранов | Промежуточные упражнения, короткие проверки на 2–4 вопроса |

Банк вопросов вкладывается во фрейм основной `vertical_scene`. Не является самостоятельной страницей.

Подробнее → [`assets/scene-types.yaml`](assets/scene-types.yaml)

### 2.4. Фрейм

Минимальная единица ритма — законченный шаг прохождения внутри сцены или банка вопросов.

**Обязательные поля:**

| Поле         | Описание                               |
| ------------ | -------------------------------------- |
| `id`         | 7-символьный hexadecimal               |
| `role`       | Роль фрейма в сценарии (см. раздел 3) |
| `completion` | Условие завершения (см. раздел 3)      |

#### Правила композиции фрейма

Фрейм может содержать `blocks`, `nested_scene` и `question_bank` одновременно — технически это допустимо. Но **рекомендуемый паттерн** — один фрейм = одна задача:

| Паттерн                                          | Рекомендация                                                                                              |
| ------------------------------------------------ | --------------------------------------------------------------------------------------------------------- |
| Фрейм с текстом + изображением                   | Нормально — блоки дополняют друг друга                                                                    |
| Вступление к банку вопросов + сам банк           | **Разделять на два фрейма**: один для вступления (`role: transition`), другой для банка (`role: practice`) |
| Текст перед вложенной сценой + сама сцена        | Допустимо — текст служит контекстом для вложенной сцены                                                  |
| Текст + вложенная сцена + банк вопросов          | **Не рекомендуется** — разбить на отдельные фреймы                                                       |

**Порядок внутри фрейма:** если фрейм содержит и блоки, и вложенную сущность — блоки идут первыми (контекст), затем `nested_scene` или `question_bank`.

**Типичный паттерн для банка вопросов:**

```
vertical_scene
  ├─ frame: transition   ← вступительный текст перед проверкой
  ├─ frame: practice     ← question_bank (серия вопросов)
  └─ frame: summary      ← фидбек / итог после проверки
```

### 2.5. Блок

Минимальная единица содержания или взаимодействия внутри фрейма. Все блоки делятся на два типа:

| Категория      | Назначение                                        | Примеры                                                                       |
| -------------- | ------------------------------------------------- | ----------------------------------------------------------------------------- |
| `presentation` | Объяснение, показ, иллюстрация, структурирование  | `tinymce`, `image`, `video`, `accordion`, `tabs`, `carousel`, `hotspot` и др. |
| `assessment`   | Проверка понимания, тренировка, рефлексия          | `single_choice`, `multiple_choice`, `sorting`, `matching` и др.               |

Полный справочник типов → [`assets/block-types.yaml`](assets/block-types.yaml)

## 3. Роли фреймов и условия завершения

`role` — семантическая метка, определяющая назначение фрейма в сценарии прохождения. Задаёт контракт для проектирования: какой контент ожидается в этом фрейме.

| Роль         | Назначение                                      | Рекомендуемый `completion`    | Типичное содержимое                                           |
| ------------ | ----------------------------------------------- | ----------------------------- | ------------------------------------------------------------- |
| `intro`      | Введение в тему, создание контекста             | `read`                        | `tinymce`, `image`, `video`                                   |
| `concept`    | Объяснение ключевых идей, теория                | `read`, `open_all_items`      | `tinymce`, `accordion`, `tabs`, `image`                       |
| `example`    | Иллюстрация на примере, демонстрация            | `read`, `click_continue`      | `tinymce`, `image`, `carousel`, вложенная `horizontal_scene`  |
| `practice`   | Проверка, упражнение, тренировка                | `pass_assessment`, `interact` | `question_bank`, assessment-блоки                             |
| `summary`    | Подведение итогов, выводы                       | `read`                        | `tinymce`, `note`, `quote`                                    |
| `reflection` | Осмысление, открытый вопрос                     | `interact`                    | `reflection`-блок, `tinymce`                                  |
| `transition` | Связка между частями, вступление к следующему шагу | `read`, `click_continue`   | `tinymce`                                                     |
| `reference`  | Дополнительные материалы, ссылки                | `read`                        | `tinymce`, `table`                                            |

**Условия завершения (`completion`):**

| Значение          | Когда фрейм считается пройденным                              |
| ----------------- | -------------------------------------------------------------- |
| `read`            | Пользователь прочитал (прокрутил / увидел)                     |
| `open_all_items`  | Пользователь раскрыл все элементы (accordion, tabs)            |
| `watch`           | Пользователь просмотрел видео                                  |
| `pass_assessment` | Пользователь ответил на все вопросы (банк / assessment-блоки)  |
| `click_continue`  | Пользователь нажал кнопку «Далее»                             |
| `interact`        | Пользователь совершил любое взаимодействие                     |

## 4. Типы блоков

### 4.1. Presentation blocks

**Базовые:**

| Тип         | Назначение                                                 |
| ----------- | ---------------------------------------------------------- |
| `rich_text` | Богатый текстовый блок (заголовки, списки, форматирование) |
| `image`     | Изображение / иллюстрация                                  |
| `video`     | Видео-вставка                                              |
| `quote`     | Цитата                                                     |
| `note`      | Заметка / выделенный блок                                  |
| `table`     | Таблица                                                    |
| `hotspot`   | Интерактивная картинка с точками                           |
| `flipcard`  | Карточки с переворотом                                     |
| `timeline`  | Временная шкала                                            |

**С вложенной структурой:**

Содержат набор элементов, внутри каждого — другие блоки. Различаются способом отображения. Не задают ритм страницы — только организуют содержание внутри фрейма.

| Тип         | Логика                            | Когда использовать                                         |
| ----------- | --------------------------------- | ---------------------------------------------------------- |
| `accordion` | Элементы раскрываются вертикально | Компактная подача нескольких независимых смысловых пунктов  |
| `tabs`      | Переключение между элементами     | Сравнение, разные роли, альтернативные точки зрения        |
| `carousel`  | Перелистывание как карточки       | Серия однотипных примеров без отслеживания прогресса        |

Внутри элементов `accordion`, `tabs`, `carousel` допускаются только базовые блоки: `tinymce`, `image`, `note`, `quote`, `table`. Вложенные блоки с собственной вложенностью запрещены (нет accordion внутри tabs и т.п.).

### 4.2. Assessment blocks

| Тип               | Назначение                                           |
| ----------------- | ---------------------------------------------------- |
| `single_choice`   | Вопрос с одним правильным ответом                    |
| `multiple_choice` | Вопрос с несколькими правильными ответами             |
| `sorting`         | Упорядочивание элементов                             |
| `matching`        | Сопоставление пар                                    |
| `sequence`        | Восстановление правильной последовательности         |
| `classify`        | Классификация по категориям                          |
| `fill_gap`        | Заполнение пропусков в тексте                        |
| `scenario`        | Сценарный вопрос с контекстом и выбором действий     |
| `case_question`   | Вопрос на основе кейса                               |
| `reflection`      | Открытый рефлексивный вопрос (без правильного ответа) |

Подробные YAML-схемы каждого типа → [`assets/block-types.yaml`](assets/block-types.yaml)

## 5. Правила вложенности

### Что может содержать каждая сущность

| Сущность                                  | Может содержать                                                                        |
| ----------------------------------------- | -------------------------------------------------------------------------------------- |
| Страница                                  | Ровно одну `vertical_scene`                                                            |
| `vertical_scene`                          | Фреймы                                                                                 |
| `horizontal_scene`                        | Фреймы (только как вложенная сцена внутри фрейма `vertical_scene`)                     |
| Фрейм                                    | Presentation-блоки, assessment-блоки, вложенную `horizontal_scene` (опц.), банк вопросов (опц.) |
| Presentation-блок с вложенностью          | Только базовые блоки внутри своих элементов                                            |
| Банк вопросов                             | Фреймы (каждый фрейм = один вопрос/упражнение)                                        |

### Ограничения

- Страница содержит **ровно одну** `vertical_scene` (не ноль, не несколько)
- Вложенные сцены — только `horizontal_scene`, только внутри фреймов `vertical_scene`
- Глубина вложенности сцен — не более 2 уровней (`vertical_scene` → `horizontal_scene`)
- Банк вопросов вкладывается во фрейм, но сам не может содержать вложенные сцены
- Банк вопросов не может содержать другой банк вопросов
- Presentation-блоки с вложенной структурой не могут содержать другие блоки с вложенной структурой
- Пустой фрейм (без блоков и без вложенных сущностей) запрещён

### Нежелательные комбинации

| Комбинация                                                              | Почему нежелательно                                     | Что делать вместо                            |
| ----------------------------------------------------------------------- | ------------------------------------------------------- | -------------------------------------------- |
| `nested_scene` + `question_bank` в одном фрейме                        | Перегружает фрейм двумя разными ритмами                  | Разбить на отдельные фреймы                  |
| `video` внутри `accordion`                                              | Скрытое видео плохо обнаруживается                       | Вынести `video` в отдельный фрейм            |
| `reflection` внутри `horizontal_question_bank`                          | Открытый вопрос в формате «быстрых карточек» сбивает ритм | Поставить `reflection` в отдельный фрейм     |
| Assessment-блок + `question_bank` в одном фрейме                        | Смешивается одиночный вопрос и серия вопросов            | Выбрать одно: либо inline-assessment, либо банк |
| Больше 2 вложенных presentation-блоков (`accordion` + `tabs`) в одном фрейме | Информационная перегрузка                            | Разнести по отдельным фреймам                |

## 6. Когда сцена, когда presentation-блок

### Сцена vs. accordion / tabs / carousel

| Критерий                                    | Сцена (`horizontal_scene`)                        | Presentation-блок (`accordion` / `tabs` / `carousel`)  |
| ------------------------------------------- | ------------------------------------------------- | ------------------------------------------------------ |
| Есть собственный прогресс / completion?      | Да — каждый фрейм отслеживается                   | Нет — просто подача внутри фрейма                       |
| Элементы — самостоятельные шаги сценария?   | Да                                                | Нет — дополнительные грани одной идеи                   |
| Пользователь должен пройти все элементы?    | Да — ритм обязывает                               | Зависит от `completion` фрейма                          |
| Нужно чередовать разные типы блоков?        | Да — в каждом фрейме свой набор                   | Нет — однородная структура                              |

**Практическое правило:** если у каждого элемента есть собственная учебная задача и условие завершения — это фреймы сцены. Если элементы раскрывают одну идею с разных сторон — это presentation-блок.

### Carousel vs. horizontal_scene

| Критерий                 | `carousel`                                    | `horizontal_scene`                                  |
| ------------------------ | --------------------------------------------- | --------------------------------------------------- |
| Отслеживание прогресса   | Нет                                           | Да — каждый фрейм с `completion`                     |
| Элементы однотипные      | Обычно да                                     | Могут быть разными                                   |
| Содержит assessment      | Нет                                           | Может                                               |
| Навигация                | Перелистывание                                | Пошаговое прохождение                               |

**Carousel** — для галереи однотипных элементов (примеры, карточки, факты). Нет прогресса, нет completion.
**Horizontal scene** — для пошагового сценария с отслеживанием. Каждый шаг = фрейм с ролью.

## 7. Учебный ритм

Ритм страницы задаётся **сценами**, **банками вопросов** и **фреймами** — через их последовательность, роли и условия завершения.

**Не задают ритм** (только организуют содержание внутри фрейма):
- accordion
- tabs
- carousel

### Типовой паттерн страницы

Самый распространённый сценарий — одна `vertical_scene` с фреймами:

```
vertical_scene
  ├─ intro       (completion: read)
  ├─ concept     (completion: read / open_all_items)
  ├─ example     (completion: click_continue)
  ├─ transition  (completion: read)            ← вступление к проверке
  ├─ practice    (completion: pass_assessment) ← question_bank
  └─ summary     (completion: read)            ← итог / фидбек
```

### Паттерны по типам обучения

| Тип обучения       | Типичная структура страницы                                                                                          | Ключевые особенности                                                 |
| ------------------- | -------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| **Compliance**      | intro → concept (accordion) → transition → practice (vertical_question_bank) → summary                              | Банк вопросов обязателен, `pass_criteria: assessment_score`           |
| **Process Change**  | intro → concept → example (horizontal_scene: walkthrough) → practice → summary                                       | Горизонтальная сцена для демонстрации нового процесса               |
| **Performance**     | intro → concept → example (carousel) → practice (scenario / case_question) → reflection → summary                    | Сценарные вопросы, reflection в финале                               |
| **Onboarding**      | intro → concept (tabs: по ролям) → example → practice → summary                                                     | Tabs для разных ролевых путей                                        |
| **Awareness**       | intro → concept → example (carousel) → summary                                                                      | Короткий формат, минимум assessment, может обойтись `reflection`     |
| **Knowledge**       | intro → concept (accordion: глубокая теория) → example → practice → summary                                         | Развёрнутая теория, гибкий assessment                                |

YAML-примеры для каждого типа → [`assets/examples-by-type.yaml`](assets/examples-by-type.yaml)

## 8. Условие зачёта страницы

Необязательное поле `pass_criteria` на уровне страницы определяет, когда страница считается пройденной.

| Значение                     | Логика                                                                     |
| ---------------------------- | -------------------------------------------------------------------------- |
| `all_frames`                 | Все фреймы с `completion` пройдены (по умолчанию)                          |
| `assessment_score: N`        | Набрано N% правильных ответов в assessment-блоках и банках вопросов         |
| `required_frames: [id1, id2]` | Пройдены конкретные обязательные фреймы (остальные — опциональные)         |

Если `pass_criteria` не указан, считается `all_frames`.

## 9. Граница модели и production-полей

Эта спецификация описывает **целевую структуру** страниц курса — то, как они выглядят в финальном виде. Она **не содержит** полей, связанных с процессом производства.

Production-поля, которые добавляются скиллами пайплайна:

| Поле                                             | Где появляется                    | Кто добавляет                                |
| ------------------------------------------------ | --------------------------------- | -------------------------------------------- |
| `status` (`empty` / `filled` / `prompt_ready`)   | На каждом блоке                   | `course-template`                            |
| `purpose`                                        | На каждом блоке                   | `course-matrix` → `course-template`          |
| `source_ref`                                     | На блоках с привязкой к исходникам | `course-matrix` → `course-template`          |
| `content_basis` (`source` / `brief` / `generate`) | На каждом блоке                   | `course-matrix` → `course-template`          |
| `content_volume` (`small` / `medium` / `large`)  | На странице                       | `course-structure` → `course-template`       |
| `_lock`                                          | На странице                       | Оркестратор (`generation-quiz`, `generation-image`) |

Скилл `course-template` берёт целевую структуру из этой модели и обогащает её production-полями для работы генерационных скиллов.

| `course-structure`  | Определяет страницы и их цели; ещё без детализации до сцен/фреймов                                    |
| `course-matrix`     | Планирует сцену и фреймы для каждой страницы, назначает типы блоков, роли, `purpose` и `content_basis` |
| `course-template`   | Создаёт YAML-файлы страниц по этой модели: сцена → фреймы → блоки с плейсхолдерами и production-полями |
| `generation-text`   | **Не читает** эту спецификацию. Работает с готовыми YAML-шаблонами из `content/pages/`                   |
| `generation-quiz`   | **Не читает** эту спецификацию. Работает с готовыми YAML-шаблонами из `content/pages/`                   |
| `generation-image`  | **Не читает** эту спецификацию. Работает с готовыми YAML-шаблонами из `content/pages/`                   |
