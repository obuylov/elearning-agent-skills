# Пайплайн производства электронных курсов

Этот проект — система для автоматизации разработки электронных курсов. Модель выполняет роль методолога и разработчика, работая совместно с пользователем.

**Формат выхода:** курс в формате лонгридов — интерактивные страницы с текстом, изображениями, интерактивными блоками и проверочными заданиями. Выбор LMS, платформы, SCORM/xAPI не входит в скоуп пайплайна.

## Структура проекта

```
sources/
  original/          ← Исходные файлы (.docx, .pdf, .pptx, .xlsx)
  converted/         ← Сконвертированные в Markdown

workspace/
  brief.md                  ← Единый бриф проекта (методология + общие параметры)
  course-structure.yaml     ← План курса: каркас + содержание (scene → frames → blocks)
  sources-report.md         ← Отчёт аудита исходников
  audit/                    ← Отчёты аудита качества (по файлу на каждую проверку)

content/
  pages/             ← YAML-файлы страниц курса (шаблоны → заполненные)
  files/             ← Изображения и медиафайлы

project-state.json        ← Статус проекта и трекинг выполнения скиллов
```

## Пайплайн: 4 кластера + 2 отдельных скилла, 14 скиллов

### Скилл инициализации

`[start](.cursor/skills/start/SKILL.md)` — инициализация проекта, проверка структуры папок, определение нового/существующего проекта. После определения `source_mode` передаёт управление кластерному агенту `analyst`. Артефакт: `project-state.json`.

### Кластер 1 — Анализ и брифование

```
start ──→ (agent: analyst) ──► analysis-scanner ──авто──→ analysis-router ──авто──→ analysis-general
                                   ↑                                                   │
                                   └──────────────────────── (ветка без исходников) ───┘
```


| Скилл / агент                                                        | Задача                                                                                                     | Ключевые артефакты                                                                         |
| -------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| `agent: analyst` (`.cursor/agents/analyst.md`)                       | Оркестрация Кластера 1: решает, запускать ли `analysis-scanner`, управляет переходом к router/general     | Обновлённый `project-state.json`, согласованный маршрут анализа                            |
| `[analysis-scanner](.cursor/skills/1_1_analysis-scanner/SKILL.md)`   | Конвертация исходников через `markitdown`, классификация контента (сырые / готовый сценарий)              | `sources/converted/*.md`, `workspace/sources-report.md`                                    |
| `[analysis-router](.cursor/skills/1_2_analysis-router/SKILL.md)`     | Определение позиции по трём осям: бизнес-драйвер × категория × аудитория                                   | Обновлённый `project-state.json` с `business_driver`, `course_category`, `target_audience` |
| `[analysis-general](.cursor/skills/1_3_analysis-general/SKILL.md)`   | Брифование по трём осям + общие параметры + финальное подтверждение, сбор и утверждение `workspace/brief` | `workspace/brief.md`                                                                       |


**Трёхосевая модель WHY + WHAT + WHO.** Каждый проект определяется по трём осям:

**Ось Y — Бизнес-драйвер (зачем):** ассеты в `assets/drivers/`


| Драйвер       | Ключевая идея                                                  |
| ------------- | -------------------------------------------------------------- |
| `compliance`  | Обязательное обучение по нормативу. Minimum viable compliance. |
| `rollout`     | Обучение под изменения. ADKAR-диагностика.                     |
| `performance` | Устранение рабочего разрыва. Диагностика Mager & Pipe.         |
| `onboarding`  | Мета-тип: декомпозиция на подмодули по категориям.             |
| `knowledge`   | Кругозор и понимание. Проверка на скрытый performance gap.     |


**Ось X — Категория курса (что):** ассеты в `assets/categories/`


| Категория    | Предметная область                             |
| ------------ | ---------------------------------------------- |
| `processes`  | Процессы и регламенты                          |
| `products`   | Продукты и услуги                              |
| `hardskills` | Hard skills (инструменты, технические навыки)  |
| `softskills` | Soft skills (коммуникация, управление, сервис) |
| `business`   | Бизнес-среда (отрасль, рынок, стратегия)       |


**Ось Z — Аудитория (для кого):** ассеты в `assets/audiences/`


| Аудитория   | Профиль                                           |
| ----------- | ------------------------------------------------- |
| `frontline` | Линейные сотрудники (массовая аудитория)          |
| `managers`  | Руководители и менеджеры                          |
| `newcomers` | Новые сотрудники (адаптация)                      |
| `experts`   | Эксперты и специалисты                            |
| `external`  | Внешняя аудитория (клиенты, партнёры, подрядчики) |


`analysis-general` загружает по одному файлу из каждой оси и комбинирует вопросы: WHY (из драйвера) + WHAT (из категории) + WHO (из аудитории) + универсальные (объём, тон, дедлайн).

Для пользователя весь кластер — непрерывное брифование. Все данные записываются в `workspace/brief.md`.

### Кластер 2 — Проектирование

```
          (agent: architect)
                 ↓
        course-model (спецификация — читается, не выполняется)
                 ↓
course-structure → course-matrix → course-template
```


| Скилл / агент                                                      | Задача                                                                                                                                                                                           | Артефакт                                                  |
| ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------- |
| `agent: architect` (`.cursor/agents/architect.md`)                 | Оркестратор Кластера 2: учитывает спецификацию `course-model`, запускает и согласует `course-structure`, `course-matrix`, `course-template`                                                     | Консистентный переход от брифа к шаблонам страниц         |
| `[course-model](.cursor/skills/2_1_course-model/SKILL.md)`         | **Спецификация** модели учебной страницы: иерархия (страница → сцена → фреймы → блоки), типы сцен, банки вопросов, правила вложенности. Читается скиллами проектирования, не generation-скиллами | Нет (референсный документ)                                |
| `[course-structure](.cursor/skills/2_2_course-structure/SKILL.md)` | Верхнеуровневый каркас: модули, страницы, learning objectives, хронометраж. Без ID, только смыслы                                                                                                | `workspace/course-structure.yaml` (каркас)                |
| `[course-matrix](.cursor/skills/2_3_course-matrix/SKILL.md)`       | Планирует содержание каждой страницы по модели: сцена → фреймы (с ролями и completion) → блоки + вложенные сцены и банки вопросов. Работает в том же файле                                       | `workspace/course-structure.yaml` (+ scene/frames/blocks) |
| `[course-template](.cursor/skills/2_4_course-template/SKILL.md)`   | Создаёт YAML-файлы страниц по модели: добавляет ID (hex), статусы, поля блоков с плейсхолдерами. Переносит метаинформацию из плана                                                               | `content/pages/*.yaml` (status: empty)                    |


`course-model` — не пошаговый процесс, а спецификация. Он определяет целевую структуру страниц, на которую опираются `course-matrix` и `course-template`. Generation-скиллы модель не читают — вся нужная информация уже заложена в YAML-файлы.

На каждом этапе возможен возврат к предыдущему скиллу или к брифу, если пользователь обнаруживает проблему.

### Кластер 3 — Генерация контента

```
        (agent: generator)
                 ↓
generation-text ──→ generation-quiz ──→ generation-image ──→ общий отчёт
```

> **Единая цепочка.** Три скилла работают без подтверждения пользователя между ними. Запуск: ручной вызов `generation-text`. Общий отчёт выводится только по завершении всех трёх этапов.


| Скилл / агент                                                      | Задача                                                                                                                    | Что заполняет / оркестрирует                                                                                                            |
| ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `agent: generator` (`.cursor/agents/generator.md`)                 | Оркестратор Кластера 3: следит за порядком и завершённостью `generation-text`, `generation-quiz`, `generation-image`     | Вся цепочка генерации (тексты, квизы, изображения)                                                                                      |
| `[generation-text](.cursor/skills/3_1_generation-text/SKILL.md)`   | Последовательная генерация текстов одним агентом, партиями по 10 страниц. Глоссарий терминов между батчами                | `tinymce`, `accordion`, `tabs`, `carousel`, `flipcard`, `timeline`, `quote`, `note`, `table`                                             |
| `[generation-quiz](.cursor/skills/3_2_generation-quiz/SKILL.md)`   | **Оркестратор:** запускает субагент `generator-quiz` в фоне для каждой страницы с assessment-блоками параллельно          | `single_choice`, `multiple_choice`, `sorting`, `matching`, `sequence`, `classify`, `fill_gap`, `scenario`, `case_question`, `reflection` |
| `[generation-image](.cursor/skills/3_3_generation-image/SKILL.md)` | **Оркестратор:** запускает субагент `generator-images` в фоне для каждой страницы с image/hotspot-блоками параллельно     | `image`, `hotspot` (все поля)                                                                                                            |


**Критическое правило:** все три скилла работают **только с файлами `content/pages/*.yaml`** — не заглядывают в `course-structure.yaml`. Вся метаинформация (purpose, source_ref, content_basis) уже встроена в блоки страниц скиллом `course-template`.

**Карта заполнения блоков:** `video` заполняется пользователем вручную. Блоки находятся внутри фреймов (scene → frames → blocks), в банках вопросов и во вложенных сценах. Каждый тип блока заполняется строго одним скиллом.

**Стратегия обработки:** `generation-text` работает последовательно (один агент), но **партиями по 10 страниц**. Между батчами обновляется глоссарий терминов (`workspace/_glossary.yaml`) — это обеспечивает консистентность даже при сбросе контекста на больших курсах. `generation-quiz` и `generation-image` — оркестраторы, запускают по одному фоновому субагенту на страницу параллельно. Прогресс трекается через `status` блоков: `empty` → `filled` / `prompt_ready`. При прерывании и повторном запуске скилл подхватывает работу с первого незаполненного блока.

**Цепочка без подтверждений:** `generation-text` по завершении автоматически запускает `generation-quiz`, тот — `generation-image`. Между этапами пользователь видит лишь краткий статус (1–2 строки). Общий отчёт по всем трём этапам выводится в конце `generation-image`. Запуск цепочки: ручной вызов `generation-text`.

| Скилл                                                        | Задача                                                                                                                                                                                                      |

### Субагенты

Субагенты — фоновые AI-ассистенты, которым скиллы делегируют параллельную работу, а также кластерные агенты, которые оркестрируют несколько скиллов. Каждый агент запускается в собственном контекстном окне и возвращает результат родительскому скиллу или команде.


| Файл                                                                       | Модель  | Режим      | Вызывается из                                                      | Назначение                                   |
| -------------------------------------------------------------------------- | ------- | ---------- | ------------------------------------------------------------------ | -------------------------------------------- |
| `[.cursor/agents/analyst.md](.cursor/agents/analyst.md)`                   | inherit | foreground | `[start](.cursor/skills/start/SKILL.md)`                           | Оркестратор кластера анализа и брифования    |
| `[.cursor/agents/architect.md](.cursor/agents/architect.md)`               | inherit | foreground | Кластеры после анализа (ручной запуск / сценарий orchestration)   | Оркестратор кластера проектирования          |
| `[.cursor/agents/generator.md](.cursor/agents/generator.md)`               | inherit | foreground | Этап генерации контента (после проектирования)                    | Оркестратор кластера генерации               |
| `[.cursor/agents/generator-quiz.md](.cursor/agents/generator-quiz.md)`     | inherit | background | `[generation-quiz](.cursor/skills/3_2_generation-quiz/SKILL.md)`   | Проверочные блоки одной страницы             |
| `[.cursor/agents/generator-images.md](.cursor/agents/generator-images.md)` | inherit | background | `[generation-image](.cursor/skills/3_3_generation-image/SKILL.md)` | Image/hotspot-блоки одной страницы           |


Скиллы `generation-quiz` и `generation-image` — оркестраторы. Запускают **по одному экземпляру субагента на каждую страницу**, где есть незаполненные блоки нужного типа. Все экземпляры работают параллельно. Субагент получает путь к файлу страницы и необходимый контекст (стратегию тестирования, бренд-палитру).

> `**generation-text` работает без субагентов** — один агент обрабатывает все страницы последовательно. Это обеспечивает единую терминологию, нарратив и стиль по всему курсу. Квизы и картинки изолированы по страницам, поэтому для них параллелизация безопасна.

| Файл                                                                                   | Модель  | Режим      | Проверка              |
| -------------------------------------------------------------------------------------- | ------- | ---------- | --------------------- |

| `[.cursor/agents/audit-brief-compliance.md](.cursor/agents/audit-brief-compliance.md)` | inherit | background | Соответствие брифу    |

Скилл `quality-audit` спрашивает пользователя: запустить все 5 проверок или конкретные. Выбранные субагенты запускаются параллельно. Каждый субагент записывает результат в свой файл в `workspace/audit/`.

> **Модели:** `audit-completeness` и `audit-technical` используют `fast` — это формальные проверки (статусы, ID, YAML-синтаксис), не требующие глубокого рассуждения. Остальные три используют `inherit` — содержательный анализ текста, методики и фактов.

## Логика переходов

### Автоматические переходы (без подтверждения пользователя)

- `[analysis-scanner](.cursor/skills/1_1_analysis-scanner/SKILL.md)` → `[analysis-router](.cursor/skills/1_2_analysis-router/SKILL.md)` — после анализа исходников
- `[analysis-router](.cursor/skills/1_2_analysis-router/SKILL.md)` → `[analysis-general](.cursor/skills/1_3_analysis-general/SKILL.md)` — после определения типа
- `[generation-text](.cursor/skills/3_1_generation-text/SKILL.md)` → `[generation-quiz](.cursor/skills/3_2_generation-quiz/SKILL.md)` → `[generation-image](.cursor/skills/3_3_generation-image/SKILL.md)` — единая цепочка генерации

### Переходы с подтверждением

Все остальные переходы требуют явного согласия пользователя. В цепочке генерации подтверждение запрашивается только в конце — после общего отчёта `generation-image`.

### Точки возврата

Скиллы `[course-structure](.cursor/skills/2_2_course-structure/SKILL.md)`, `[course-matrix](.cursor/skills/2_3_course-matrix/SKILL.md)`, `[course-template](.cursor/skills/2_4_course-template/SKILL.md)` позволяют вернуться на предыдущий этап без обновления `project-state.json`.

Финальные файлы курса: YAML-страницы с контентом и медиафайлы. Сначала создаются пустые шаблоны, затем наполняются контентом, затем проходят аудит.

## Режимы работы с исходниками

Исходные материалы могут быть в разном объёме — от полного набора до полного отсутствия. Пайплайн работает во всех трёх режимах:


| Режим       | Описание                           | Откуда берётся контент                                                                                                            |
| ----------- | ---------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| **full**    | Исходники покрывают всю тему курса | Из `sources/converted/` — переработка и адаптация                                                                                 |
| **partial** | Исходники покрывают тему частично  | Где есть `source_ref` → из исходников. Где нет → генерация на основе брифа, домейн-знаний агента и уточнений у пользователя       |
| **none**    | Исходников нет совсем              | Всё генерируется на основе брифа и экспертизы пользователя. Агент ведёт расширенное брифование, задаёт больше уточняющих вопросов |


Режим фиксируется в `project-state.json` (поле `source_mode`) и в `brief.md` (поле `source_coverage`). Скиллы ниже по пайплайну читают этот режим и адаптируют поведение.

**Если режим `none` или `partial`:**

- `course-matrix` ставит `source_ref: null` для блоков без исходника и `content_basis: brief` или `generate`
- `course-template` переносит эту метаинформацию в файлы страниц — generation-скиллы работают автономно
- `generation-text` генерирует контент на основе `purpose` блока, брифа и domain knowledge
- `quality-audit` не проверяет «соответствие исходникам» для сгенерированных блоков, но проверяет фактическую корректность и соответствие брифу

## Статусы блоков в content/pages/


| Статус         | Значение                                                                      |
| -------------- | ----------------------------------------------------------------------------- |
| `empty`        | Плейсхолдер, контент не сгенерирован                                          |
| `filled`       | Контент заполнен                                                              |
| `prompt_ready` | Для image-блоков: промт для генерации подготовлен, изображение ещё не создано |


## Транзакционная дисциплина

Параллельная работа субагентов требует защиты от гонок записи, конфликтов и частичных результатов.

### `_active` — блокировка на уровне project-state.json

Перед началом обработки страниц скилл устанавливает в `project-state.json`:

```yaml
_active:
  skill: generation-quiz
  started: "2026-03-29T14:00:00"
  pages_total: 10
  pages_done: 0
```

- Любой другой скилл на Шаге 0 видит `_active` → предупреждает пользователя и не начинает без подтверждения.
- `edit` при обнаружении `_active` предлагает подождать.
- После завершения работы скилл удаляет `_active`.

### `_lock` — блокировка на уровне страницы

Оркестратор перед запуском субагента добавляет в YAML страницы:

```yaml
_lock:
  agent: generator-quiz
  started: "2026-03-29T14:05:00"
```

- Субагент не трогает `_lock` — снимает только оркестратор после валидации результата.
- `edit` отказывается редактировать страницу с `_lock`.

### Жизненный цикл оркестратора

```
1. Установить _active в project-state.json
2. Для каждой страницы:
   a. Установить _lock в файле страницы
   b. Запустить субагент в фоне
3. Дождаться завершения всех субагентов
4. Для каждой страницы:
   a. Валидировать результат (YAML парсится, целевые блоки filled)
   b. Если валидация не пройдена → перезапустить субагент (1 попытка)
   c. Снять _lock
   d. Обновить _active.pages_done
5. Удалить _active из project-state.json
6. Обновить статус скилла → completed
```

### Восстановление после сбоя

При повторном запуске скилл обнаруживает `_active` от предыдущего прерванного запуска:

1. Снимает все устаревшие `_lock` на страницах
2. Составляет `pages_to_process` из страниц с `empty`-блоками
3. Запускает субагенты только для них

Идемпотентность обеспечивается тремя точками состояния: `_active` → `_lock` → `block.status`.

## Общие правила для всех скиллов

1. **Шаг 0** каждого скилла — прочитать `project-state.json`, проверить `_active`, убедиться, что предыдущий этап выполнен.
2. **Последний шаг** — удалить `_active` (если устанавливался), обновить `project-state.json`, отметив свой этап как выполненный.
3. Все скиллы работают с файлами в рамках структуры проекта — не создают файлы за её пределами.
4. Если есть исходники (`source_ref`) — контент основывается на них. Если исходников нет — контент генерируется на основе брифа, domain knowledge и уточнений у пользователя. В обоих случаях не выдумывать факты, цифры и имена.
